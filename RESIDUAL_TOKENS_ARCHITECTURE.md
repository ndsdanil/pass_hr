# Архитектура системы возврата остаточных токенов

## 🎯 **Правильная архитектура**

Система возврата токенов работает по **двум разным путям**, и это правильно:

### 📊 **Два пути обработки:**

#### 1. **RabbitMQ путь** (`use_rabbitmq=True`)
```
HTTP API → RabbitMQ Queue → Resume-Tuner → Results Queue → RabbitMQ Consumer → Возврат токенов
```

#### 2. **HTTP путь** (`use_rabbitmq=False`)
```
HTTP API → Resume-Tuner HTTP → HTTP API → Возврат токенов
```

## 🏗️ **Компоненты системы**

### 1. **HTTP API** (`backend/app/api/v1/resume.py`)
- **Для `use_rabbitmq=True`**: Только отправляет в очередь
- **Для `use_rabbitmq=False`**: Обрабатывает результат И возвращает токены

### 2. **RabbitMQ Consumer** (`backend/app/services/rabbitmq_consumer.py`)
- **Только для `use_rabbitmq=True`**: Обрабатывает результаты из очереди
- Возвращает токены пользователю
- Сохраняет метрики в БД

### 3. **BillingService** (`backend/app/services/billing.py`)
- **Централизованная логика** работы с токенами
- Используется в обоих путях

## ✅ **Почему это правильно?**

### 🔄 **Разные пути = разная обработка**
1. **RabbitMQ**: Асинхронная обработка, результат приходит через очередь
2. **HTTP**: Синхронная обработка, результат приходит сразу

### 📍 **Единая точка логики**
- Вся логика работы с токенами в `BillingService.add_residual_tokens()`
- Разные пути используют один и тот же метод

### 🎯 **Правильное разделение ответственности**
- **HTTP API**: Обрабатывает синхронные запросы
- **RabbitMQ Consumer**: Обрабатывает асинхронные результаты

## 📝 **Логирование для отладки**

### HTTP API:
```
INFO - Возвращено 9.997 остаточных токенов пользователю 123 (HTTP API)
```

### RabbitMQ Consumer:
```
INFO - Добавлено 9.997 остаточных токенов пользователю 123
INFO - Возвращено 9.997 остаточных токенов пользователю 123
```

## 🔍 **Как определить путь обработки?**

### В логах HTTP API:
- Если `use_rabbitmq=True`: Только отправка в очередь
- Если `use_rabbitmq=False`: Полная обработка + возврат токенов

### В логах RabbitMQ Consumer:
- Всегда обрабатывает результаты из очереди
- Всегда возвращает токены

## ⚠️ **Важные моменты**

1. **Не дублирование**: Это разные пути для разных сценариев
2. **Централизация**: Логика токенов только в `BillingService`
3. **Консистентность**: Одинаковое поведение в обоих путях
4. **Отслеживание**: Метрики сохраняются в обоих случаях
